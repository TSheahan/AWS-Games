# ec2/minecraft/ â€” EC2-Side Minecraft Scripts

Scripts in this directory run on the **EC2 instance**. `ec2/minecraft/setup.sh` is invoked by UserData during first-boot provisioning; it handles instance-level setup (JDK, JAR download) then delegates all server-specific work to `provision_servers.py`. Start/stop scripts and systemd units are generated by `provision_servers.py` into each server's working directory under `/mnt/persist/minecraft/<folder>/`.

`setup.sh` is classified at the `minecraft/` level by **content scope** â€” its content is entirely Minecraft-specific (JDK version selection, `/mnt/persist/minecraft/` paths). Invocation via UserData (a singleton per instance lifecycle) does not make it game-agnostic. The CloudFormation `SetupCommand` parameter is the game-selection mechanism: it carries the path to whichever game's setup script is needed. A future game type would supply `ec2/<game>/setup.sh` alongside this directory.

**Target environment:** Amazon Linux 2023, ARM64 (Graviton). Do not introduce x86-specific packages, paths, or assumptions.

## Start/stop scripts

Generated per-server by `provision_servers.py` into `/mnt/persist/minecraft/<folder>/`. Not checked in as repo files â€” `provision_servers.py` is the source of truth for their content.

**start-minecraft.sh** (generated):
- Pre-launch validation: `server.properties` and `eula=true` in `eula.txt` (exits non-zero if either fails)
- Screen session name: `minecraft-<server_id>`
- Console output redirected to `console_YYYY-MM-DD_HH-MM-SS.log` in the server folder
- JVM flags: `-Xmx4092M -Xms4092M -Djava.net.preferIPv4Stack=true` (written by `provision_servers.py`)

**stop-minecraft.sh** (generated):
- Guards against no active screen session (exits 0 if server already stopped)
- `/say` at 20s â†’ 10s sleep â†’ `/say` at 10s â†’ 10s sleep â†’ `/stop` (20s total pre-stop elapsed)
- `TimeoutStopSec=90` in generated systemd units (70s remaining after `/stop` for chunk saves)

## Systemd integration

All units are generated by `provision_servers.py` as `minecraft-<server_id>.service`. Stale units for servers removed from config are cleaned up automatically on each provision run.

Attach to a running console: `screen -r minecraft-<server_id>`
Detach without stopping: `Ctrl+A, D`

Follow live server logs: `sudo journalctl -u minecraft-<server_id> -f`
Last N lines: `sudo journalctl -u minecraft-<server_id> -n 100`

## `provision_servers.py`

Multi-server systemd provisioner.

**Runs on:** EC2 instance, as root
**Config source:** `https://github.com/TSheahan/AWS-Games-Config.git`
**Port bounds source:** `/home/ec2-user/game-ports.json` (written by CloudFormation UserData)
**Log file:** `/var/log/minecraft-provision.log` (world-readable)

Modes (use together or separately):
- `--update` â€” clone or pull the config repo
- `--read-only` â€” validate config without writing any files
- `--provision` â€” full write: systemd units, start/stop scripts, server.properties, enabled/disabled state

Generates per-server files under `/mnt/persist/minecraft/<folder>/` and systemd units under `/etc/systemd/system/minecraft-<server_id>.service`. Cleans up stale units for servers removed from config.

## `minecraft` admin wrapper

Installed by `setup.sh` to `/home/ec2-user/bin/minecraft` (on PATH). Bash completion
drop-in installed to `/etc/bash_completion.d/minecraft`.

**Subcommands:** `status [instance] [--yaml]`, `start [instance]`, `stop [instance]`,
`screen <instance>`, `reprovision`

`status` without `--yaml` prints an emoji-annotated table. `--yaml` emits machine-readable
output suitable for agent consumption. `reprovision` calls `provision_servers.py` via sudo.

## Design principle: provision_servers.py owns all server-specific setup

`setup.sh` handles instance-level concerns only: JDK install, JAR download to
`/mnt/persist/minecraft/server_<version>.jar`. All server-specific work â€” folder creation,
systemd units, start/stop scripts, `server.properties` â€” is owned by `provision_servers.py`,
which `setup.sh` invokes at the end (`--update --provision`).

Where behaviour overlaps (e.g. `TimeoutStopSec`), `provision_servers.py` is the reference.
